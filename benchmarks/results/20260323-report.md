## Root Cause Analysis

### Why is cache hit rate only 8.4%?

**Primary cause: LLM-generated parameters are non-deterministic.** Cache keys are an exact hash of `{tool_name, params}`. Even when two agents read the same file, their `offset`/`limit` parameters differ:

**Example — S001 (0 hits, both agents read `runtime.py`):**
```
sub0: read(file_path="runtime.py")                         → default offset=1, limit=200
sub0: read(file_path="runtime.py", offset=201, limit=100)  → continuation
sub1: read(file_path="runtime.py", offset=1, limit=400)    → reads whole file in one call
```
Same file, different params → different cache keys → 0 hits.

**Example — S092 (4 hits, both agents read `dedup.py` + `shared_memory.py`):**
```
sub1: read(dedup.py)         → MISS (executes first, stores in cache)
sub0: read(dedup.py)         → HIT ✓ (identical default params)
sub0: read(dedup.py, offset=200, limit=180)  → stored
sub1: read(dedup.py, offset=200, limit=180)  → HIT ✓ (identical params by coincidence)
```
Hits only occur when agents happen to choose the exact same parameters.

### Why is AgentGlue slower?

Three compounding factors:

1. **Prompt overhead (+15% tokens):** `AGENTGLUE_PROMPT_PREFIX` adds ~100 tokens per agent instructing it to use `agentglue_cached_*` tools. With 67 sub-agents, this is ~6,700 extra input tokens, plus the system prompt now includes proxy tool schemas alongside built-in tools.

2. **HTTP round-trip overhead:** Every proxy tool call makes 1-3 HTTP requests to the sidecar:
   - Cache check (`/cache/check`) — ~2ms
   - Sidecar execution (`/call`) — ~3ms
   - Cache store (`/cache/store`) — ~1ms

   At 203 cacheable tool calls, this adds ~1.2s of pure network overhead. Small, but combines with other factors.

3. **More tool calls (+28%):** Agents in agentglue mode made 206 tool calls vs 161 in baseline. The proxy tools may return slightly different output formats (e.g., `File: path\nLines: X-Y of Z\n===\n...` vs the built-in `read` format), causing the LLM to make additional calls for context it would have gotten in a single built-in call.

### Why do some tasks show speedup without cache hits?

Tasks like S012 (1.63x), S013 (1.91x), S073 (2.14x) show significant speedup with 0 cache hits. This is **LLM behavioral variance** — the model simply chose a more efficient strategy in one run vs the other. This noise is inherent to single-run benchmarks and would average out over multiple runs.

## Cache Hit Pattern Analysis

### Tasks with hits (10/30)

All 10 tasks that achieved cache hits share one characteristic: **agents used default parameters** (no `offset`/`limit` specified) for at least some calls. When `read(file_path=X)` is called without explicit offset/limit, the cache key is deterministic.

| Category | Tasks with hits | Pattern |
|----------|----------------|---------|
| same-file-different-questions | S003 | 1 hit — one default-param read overlapped |
| same-search-different-scopes | S011 | 1 hit — one overlapping search |
| duplicate-review | S041, S042, S043 | 1 hit each — same file, default params |
| multi-file-summary | S061, S062, S063 | 2 hits each — `__init__.py` read by all 3 agents |
| full-overlap-stress | S091, S092 | 2-4 hits — multiple files with matching params |

### Tasks with 0 hits (20/30)

These tasks had 0 hits because:
- **Different params for same file** (S001, S002): agents chose different `offset`/`limit`
- **Different search paths** (S012, S013): `path="/src/agentglue/core/"` vs `path="/src/agentglue/"`
- **Different search patterns** (S051-S053): each agent uses different regex
- **Different files entirely** (S021-S023, S071-S073, S081-S083): agents read different files as designed

## Conclusions

### What works
- Cache infrastructure is reliable: flush, store, check, invalidate all function correctly
- Sidecar metrics are accurate and match transcript analysis
- Cross-process SQLite cache survives across sub-agent sessions
- Single-flight coalescing is available (0 coalesced in this run — agents rarely call at the exact same millisecond)

### What doesn't work
- **Exact-match caching is fundamentally mismatched with LLM non-determinism.** LLMs choose different `offset`/`limit` for the same file, different search paths, and different regex patterns. An 8.4% hit rate does not justify the overhead.
- **Proxy tool approach adds friction:** extra prompt, extra tool schemas, extra HTTP, slightly different output format → more tokens, more time.

### Recommended next steps

1. **Fuzzy cache keys for `read`:** Hash only `file_path`, ignore `offset`/`limit`. Cache the full file content. When a cache hit occurs, extract the requested line range from the cached full content. This would have turned most 0-hit tasks into high-hit tasks.

2. **`before_tool_call` hook (proposed):** Intercept built-in `read`/`grep`/`glob` transparently — no proxy tools, no prompt injection, no parameter mismatch. This is the clean solution but requires an OpenClaw SDK change (proposal in `docs/before-tool-call-proposal.md`).

3. **Content-addressable caching for search:** For `grep`, hash only `{pattern, path}` and ignore formatting params (`-n`, `-A`, `-B`, `output_mode`). The actual search results are the same regardless of output formatting.

4. **Multi-run averaging:** Run the benchmark 3-5x and average to filter out LLM behavioral variance. Single-run speedup/slowdown numbers are unreliable.

5. **Drop `AGENTGLUE_PROMPT_PREFIX`:** Use `tools.deny: ["read", "grep", "glob"]` in `openclaw.json` instead. This eliminates prompt overhead and forces all agents (including sub-agents) through proxy tools without extra instructions.
